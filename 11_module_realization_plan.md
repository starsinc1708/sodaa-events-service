# Дополнительная часть: модуль обработки результатов поставщика

## Общая идея

Внешний поставщик асинхронно возвращает результат обработки сигнала через Kafka.

**Требования:**

- сохранить raw-событие без потерь;
- защититься от дублей сообщений;
- проверить результат по blacklist;
- сформировать очищенную запись (clean row);
- списать баланс только один раз за валидный результат;
- обеспечить устойчивость к повторным доставкам Kafka.

**Подход:** событийный конвейер с идемпотентной обработкой, PostgreSQL как источник истины.

---

## Компоненты системы

### 1. Receiver - приём сырых событий

Отдельный Kafka-consumer. Получает сообщение поставщика и сохраняет его в `raw_events`.

Назначение:
- сохранить исходный ответ для аудита;
- иметь возможность переиграть обработку;
- не потерять данные даже при ошибке дальнейших этапов.

```sql
raw_events
----------
id               UUID
provider_event_id TEXT
payload          JSONB
received_at      TIMESTAMPTZ
```

### 2. Processing Worker - бизнес-обработка

Фоновый worker выполняет обработку в несколько этапов:

1. проверка дублей;
2. проверка blacklist;
3. формирование clean row;
4. списание баланса;
5. фиксация результата.

Worker - единственное место, где принимается решение о списании средств.

---

## Идемпотентность

Kafka гарантирует только at-least-once доставку - любое сообщение может прийти повторно.

Защита через уникальный бизнес-идентификатор поставщика `provider_event_id` и таблицу обработанных событий:

```sql
processed_events
----------------
provider_event_id TEXT PRIMARY KEY
processed_at      TIMESTAMPTZ
```

Алгоритм:

```sql
INSERT INTO processed_events (provider_event_id)
VALUES (...)
ON CONFLICT DO NOTHING;
```

Если вставка не произошла (`rows_affected = 0`) - событие уже обрабатывалось, безопасно пропускаем. Повторная доставка Kafka не приводит к повторной обработке.

---

## Blacklist

Хранится в PostgreSQL:

```sql
blacklist
---------
entity_id   TEXT
reason      TEXT
created_at  TIMESTAMPTZ
```

Worker проверяет blacklist до любых финансовых операций. Если запись найдена:
- событие помечается как `rejected`;
- clean row не создаётся;
- баланс не списывается.

---

## Clean row

После прохождения всех проверок создаётся очищенная запись:

```sql
clean_results
-------------
id                  UUID
provider_event_id   TEXT
user_id             UUID
normalized_payload  JSONB
created_at          TIMESTAMPTZ
```

В clean row попадают только валидные данные, необходимые для аналитики или биллинга.

Raw и clean данные хранятся **раздельно**. Это позволяет:
- переобрабатывать историю;
- менять правила очистки;
- проводить аудит.

---

## Списание баланса

Наиболее критичная часть - защита от двойного списания. Списание выполняется внутри одной PostgreSQL-транзакции вместе с фиксацией факта обработки:

```sql
BEGIN;

INSERT INTO processed_events (...)
ON CONFLICT DO NOTHING;
-- если вставка не произошла => ROLLBACK

INSERT INTO billing_transactions (...);

UPDATE accounts
SET balance = balance - :cost
WHERE account_id = :id
  AND balance >= :cost;

INSERT INTO clean_results (...);

COMMIT;
```

### Дополнительный барьер защиты

История операций имеет уникальный ключ:

```sql
billing_transactions
--------------------
id                  UUID
provider_event_id   TEXT UNIQUE
account_id          UUID
amount              DECIMAL
created_at          TIMESTAMPTZ
```

`UNIQUE(provider_event_id)` не позволит создать вторую транзакцию списания, даже при повторной доставке. Таблица транзакций - дополнительный барьер защиты.

## Работа с Kafka

Тот же подход, что и в основном задании.

### Успешная обработка

```
BEGIN => обработка => COMMIT БД => commit offset Kafka
```

Offset коммитится только после успешного коммита транзакции БД - потеря данных исключена.

### Retry

Ошибки делятся на две группы:

**Транзиентные:** PostgreSQL недоступен, временная ошибка сети, timeout.
Сообщение не коммитится - Kafka доставит повторно.

**Неисправимые:** битый JSON, отсутствует обязательное поле, нарушен контракт поставщика.
После нескольких попыток отправляются в DLQ.

### Dead Letter Queue

Отдельный Kafka-топик `provider-results-dlq`. Сообщение дополняется контекстом:

```json
{
  "event": { "...исходное сообщение..." },
  "error": "описание ошибки",
  "attempts": 5,
  "failed_at": "2026-06-17T12:00:00Z"
}
```

Это позволяет расследовать проблемы, вручную переигрывать события и не блокировать основной поток.

## Какие тесты я бы написал

### Unit-тесты

**Deduplication:**

- первое сообщение обрабатывается;
- повторное сообщение пропускается.

**Blacklist:**

- blacklist найден => `rejected`;
- blacklist отсутствует => продолжение обработки.

**Billing:**

- корректное списание;
- отказ при недостатке средств;
- отсутствие повторного списания.

**Retry logic:**

- временная ошибка => retry;
- постоянная ошибка => DLQ.

## Основные риски и вопросы

**Возможен ли refund?**
Если поставщик позже отзывает результат, понадобится механизм компенсационных транзакций.

**Каков объём blacklist?**
Если список очень большой, возможно потребуется Redis-кэш или отдельный сервис проверки.

**Какой SLA по обработке?**
От этого зависит выбор между синхронным и полностью асинхронным конвейером.

---

## Итог

В production я бы рассматривал PostgreSQL как источник истины для raw-событий, обработанных событий и биллинга, Kafka - как транспорт доставки, а идемпотентность строил бы на уникальном бизнес-ключе поставщика и атомарной транзакции, объединяющей обработку результата и списание средств. Это позволяет безопасно работать в модели at-least-once и гарантировать отсутствие повторного списания даже при повторных доставках сообщений или перезапуске сервисов.
