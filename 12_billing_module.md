# Дополнительная часть: модуль биллинга

## Контекст

В системе есть модуль биллинга с простой предоплатной моделью (MVP):

1. У клиента есть внутренний баланс.
2. Баланс пополняется вручную через административную часть.
3. За обработанные валидные результаты с баланса списывается стоимость.
4. Списание не должно происходить за дубли, ошибки обработки или невалидные результаты.
5. По всем операциям сохраняется история транзакций.

Онлайн-оплата через платёжный шлюз не реализуется. Задача: описать перевод модуля с ручного пополнения на работу со шлюзом.

## Пересечения с модулем обработки результатов

Задача пересекается с [предыдущей](11_module_realization_plan.md) в плане:

- идемпотентности;
- надёжности;
- эволюции системы;
- понимания границ сервисов.

Я описываю архитектуру под **разные** платёжные шлюзы, а не интеграцию с одним конкретным провайдером.

## Вопросы перед проектированием

- Нужна ли поддержка нескольких платёжных провайдеров?
- Возможна ли смена провайдера без миграции данных?
- Нужны ли возвраты (refund) и частичные возвраты?
- Нужны ли рекуррентные платежи и сохранение карт?

Для MVP ответы: один провайдер, только пополнение, без автосписаний, без сохранения карт. Решение можно сильно упростить.

## Модель интеграции

Практически все провайдеры работают по одной схеме:

```
Create Payment => пользователь оплачивает => webhook от провайдера => пополнение баланса
```

Документация российских эквайрингов прямо рекомендует HTTP-уведомления (webhook) как основной способ получения статуса платежа.

## Архитектура

Я бы **не** проектировал отдельный Payment Service. Вместо этого выделяю два модуля внутри системы.

### Текущая схема (MVP)

```
Admin => Billing
```

### После подключения шлюза

```
Payment Provider => Webhook => Payment Module => Kafka => Billing
```

Billing ничего не знает о конкретном провайдере и продолжает работать только с внутренними операциями пополнения и списания. Billing остаётся источником истины, платёжный шлюз - внешний адаптер. Все денежные операции идемпотентны, надёжность обеспечивается через webhook + Kafka + reconciliation job.

## Компоненты системы

### Billing

Отвечает за:

- хранение баланса клиента;
- списание и зачисление средств;
- историю финансовых операций.

```sql
accounts
--------
account_id   UUID PRIMARY KEY
balance      DECIMAL NOT NULL
updated_at   TIMESTAMPTZ

transactions
------------
id            UUID PRIMARY KEY
account_id    UUID NOT NULL
amount        DECIMAL NOT NULL
type          TEXT NOT NULL        -- 'credit' | 'debit' | 'refund'
reference_id  TEXT UNIQUE
created_at    TIMESTAMPTZ
```

### Payment Module

Отвечает за интеграцию с платёжным провайдером:

- создание платежа;
- приём webhook-уведомлений;
- проверка подписи уведомлений;
- публикация событий пополнения баланса в Kafka.

```sql
payments
--------
id                   UUID PRIMARY KEY
account_id           UUID NOT NULL
provider             TEXT NOT NULL
provider_payment_id  TEXT NOT NULL
amount               DECIMAL NOT NULL
status               TEXT NOT NULL        -- 'pending' | 'paid' | 'failed' | 'cancelled' | 'refunded'
created_at           TIMESTAMPTZ
updated_at           TIMESTAMPTZ

UNIQUE(provider, provider_payment_id)
```

---

## Процесс пополнения баланса

1. Пользователь инициирует пополнение.
2. Система создаёт запись `payments` со статусом `pending`, получает ссылку на оплату от провайдера.
3. После оплаты провайдер отправляет webhook.
4. **Webhook не изменяет баланс напрямую.** Он только:
   - проверяет подпись запроса;
   - сохраняет информацию о платеже;
   - публикует событие `payment.completed` в Kafka.
5. Отдельный worker выполняет зачисление средств в Billing.

Такой подход изолирует внешнюю интеграцию от финансовой логики и использует уже существующую событийную инфраструктуру.

## Идемпотентность

### Защита от повторных webhook

Провайдеры могут повторно отправлять уведомления при отсутствии корректного ответа или сетевых сбоях.

```sql
processed_webhooks
------------------
provider            TEXT NOT NULL
provider_event_id   TEXT NOT NULL
processed_at        TIMESTAMPTZ

UNIQUE(provider, provider_event_id)
```

Алгоритм:

```sql
INSERT INTO processed_webhooks (provider, provider_event_id)
VALUES (...)
ON CONFLICT DO NOTHING;
```

Если запись уже существует - webhook обработан, повторная бизнес-логика не запускается.

### Защита от повторного зачисления

Одной проверки webhook недостаточно. Дополнительно каждая финансовая операция имеет уникальный внешний идентификатор:

```sql
transactions.reference_id UNIQUE   -- = provider_payment_id
```

Даже если событие повторно попадёт в Kafka, вторая операция пополнения не будет создана. Два уровня защиты:

1. идемпотентность webhook (`processed_webhooks`);
2. идемпотентность финансовой транзакции (`reference_id UNIQUE`).

## Retry и DLQ

Тот же подход, что и в основном задании.

**Транзиентные ошибки:** PostgreSQL недоступен, сетевой сбой, временная недоступность Kafka. Сообщение не коммитится - будет обработано повторно.

**Неисправимые ошибки:** несуществующий аккаунт, повреждённые данные, нарушение контракта. После нескольких попыток отправляются в `billing-topup-dlq`.

---

## Сверка платежей (Reconciliation)

Webhook нельзя считать абсолютным источником истины. Возможны ситуации: уведомление потеряно, обработано частично, сбой во время обработки.

Периодический reconciliation job выполняет сверку:

```
Payment Module => API платёжного провайдера => сверка статусов
```

Если обнаружен платёж со статусом `paid`, не приведший к зачислению, система безопасно переобрабатывает его благодаря идемпотентности операций.

---

## Итог

1. **Billing не знает о провайдере.** Payment Module - прослойка между шлюзом и финансовым ядром.
2. **Webhook не меняет баланс.** Только фиксирует факт оплаты и публикует событие в Kafka.
3. **Два уровня идемпотентности:** `processed_webhooks` + `transactions.reference_id UNIQUE`.
4. **Reconciliation job** закрывает риск потерянных webhook-уведомлений.
5. При дальнейшем росте систему можно перевести на transactional outbox/CDC без изменения бизнес-логики.
